Benchmark process

  1. Read benchmark/config/compare_backends.yaml for run definitions and dataset names.
  2. Load each dataset from the configured benchmark_root (default: sibling ../ner-dataset/datasets/<name>.jsonl) or NER_DATASET_DIR.
  3. For each (run, dataset) pair: repeat 1 time(s) — clear backend cache, run all examples, score spans, record wall-clock latency (model load + inference each trial).
  4. Aggregate trials: mean ± std latency; verify score reproducibility across repeats.
  5. Write metrics.json, report.md, and this report.html.

CLI: uv run python benchmark/run_benchmark.py — flags --pattern-only, --datasets, --runs, --max-examples, --output.

Gold data format

JSONL lines with id, text, and entities (each entity: text, label, start, end character offsets). Predictions without resolvable character offsets in the source text are skipped and counted in skipped_predictions.

arxiv_gold
ML paper abstracts with scientific entity types (model, dataset, benchmark, metric, …). bert-conll is excluded via per-run datasets in YAML (CoNLL labels only). Use GLiNER (or pattern for numbers) with matching label strings.
synthetic_*
Procedural gold corpora (news, blog, scientific, transcript, mixed) with unified person/organization/location/date labels — good for BERT and GLiNER baselines.

Scoring metrics

Document-level string overlap F1 (Doc F1)

Objective. Did the backend recover the same salient concepts as gold, regardless of how many times each string appears in the text?

How it is computed. Per example, form sets of (label, lowercased text) from gold spans and predictions (after label_maps.yaml). TP = set intersection size; FP = predictions not in gold; FN = gold not in predictions. Micro-averaged precision, recall, and F1 across all examples. Repeated spans with the same label and surface string count once.

Leaderboard. Global and per-dataset tables are sorted by Doc F1 (higher is better). Rank #1 row is highlighted in green. Use this metric when gold marks salient entities (e.g. arxiv_gold) rather than every token-level mention.

Strict span F1

Objective. Exact entity detection: same unified label and identical character span (start, end) as gold.

How it is computed. Greedy one-to-one matching per example: each prediction matches at most one gold span with the same label and exact offsets; unmatched predictions are FP, unmatched gold are FN. Micro-averaged precision, recall, and F1 across all examples in the run.

Use. Secondary table in the markdown report; penalizes extra mentions of the same entity when gold annotates a single span.

Relaxed span F1

Objective. Partial credit when boundaries differ slightly but the entity is largely correct.

How it is computed. Same label required; spans match if intersection ÷ union (IoU) ≥ 0.5. Also micro-averaged across the run.

Use. Compare backends when tokenization or whitespace causes small boundary shifts; do not treat relaxed F1 as “good enough” for production without manual review.

Latency (ms / example)

Objective. Wall-clock time for the entire backend pass over the dataset, divided by number of scored examples.

Includes. Model load (first call in that run), inference, and Python overhead. Excludes. Other backends’ runs; no warm-up run is performed separately.

Caveats. Small datasets exaggerate load time; CPU vs GPU and batching differ by backend. Use for rough operational comparison on the same machine, not absolute SLA numbers.

Label confusion matrices

Per backend run (global and each dataset section). Rows are gold labels; columns are predicted labels after label_maps.yaml normalization. Spans are paired greedily (relaxed: IoU ≥ 0.5; strict: exact offsets). Diagonal cells = correct label at a paired span. Column ∅ missed = gold with no paired prediction; row ∅ spurious = prediction with no paired gold. Use relaxed matrices to diagnose label confusion; compare with strict span pairing to separate boundary from type errors.

Radar chart

Per dataset only. Three axes — Doc F1, strict span F1, and Speed — on a 0–1 scale (higher is better). Quality metrics use raw scores. Speed uses a fixed reference: 1 − (ms/example ÷ 1000), clamped to 0–1, so one second per example maps to 0 on the Speed axis (not relative to the slowest backend in the chart).

Absolute ms/example remains in the leaderboard table. Polygon area is a visual composite index only; it weights all axes equally and is not a substitute for reading Doc F1 and latency separately.

Label normalization

Backend labels (e.g. CoNLL PER, GLiNER person) are mapped through label_maps.yaml before scoring. Comparing F1 across datasets with incompatible label schemes is misleading — e.g. do not rank pattern on CoNLL using Doc F1 alone.

How to use this scorecard

  • Pick the dataset that matches your deployment labels and language.
  • Prefer Doc F1 for salient-entity / concept extraction; use strict F1 when every span offset must match gold; use relaxed F1 to diagnose boundary issues.
  • Read latency from the leaderboard table (absolute ms/example).
  • Use label confusion matrices to see which gold types are missed or mis-tagged.
  • Treat pattern as a fast regex baseline, not a ceiling for ML quality.
